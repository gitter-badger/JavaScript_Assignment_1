Chapter 01 - Basics of JavaScript

What are some cross-browser issues that affect JavaScript code?
Answer: Web browsers rely on two distinct engines to load and render a webpage.
  • JavaScript Engine
  • Rendering Engine
The inconsistent implementation of web standards by various browser vendors of these engines make it nearly impossible to be absolutely sure if the code we author will behave and appear the same way on every platform. The DOM API provided by each of these vendors differs ever so slightly making it difficult to write simple JavaScript code without having to include shims and polyfills or some sort of fallback mechanism
What tools are available to deal with these issues?
Answer: It serves us well in general to follow these guidelines to get around these cross-browser issues
  • Don't leave anything to the browser to make assumptions. Instead be explicit wherever possible. Ex: include the DOCTYPE for every webpage
  • Ensure the HTML document is valid and is wellformed
  • Rely on popular JavaScript libraries and CSS frameworks that normalize the cross-browser idiosyncrasies.
  • Use popular unit-testing tools such as Jasmine, mocha-chai to run tests in all the browsers using a testrunner such as Karma
What considerations should we take when determining what browsers to support?
Answer: Although various strategies exist, the one that Yahoo follows seems to be more popular. It involves putting together a matrix which Yahoo calls the "graded browser matrix" where the browsers are along one axis and the platforms are along the second axis. We then need to grade them depending on the project requirements
What sort of testing should we do on JavaScript code before deployment?
Answer: Before deploying the JavaScript code, we should be certain that the test cases we author all pass and posses the three main characteristics, namely
  1. Repeatability
  2. Simplicity
  3. Isolation/Independent
Types of testing would include
  • functionality
  • performance
  • compatibility
, There are two approaches to testing the code we write when we bump into a bug and we want to debug it.
  1. Constructive
  2. Deconstructive

02 - Testing Frameworks

What debug tools can we use for JavaScript?
Answer: We can use google chrome developer tools to place break points and perform stepwise execution to debug JavaScript. Also we could use the console.log() method to log to the console.
Explain how deconstructive and constructive test cases work.
Answer: Deconstructive test cases are composed by isolating just the code that constitute the system-under-test and eleminating the rest of the code that do not participate in the test cases. Constructive test cases on the other hand are put together by starting from a known good reduced case and go on adding code to it until a bug shows up.
What is the purpose of a testing framework?
Answer: A good testing framework enables us to organize our tests by grouping them together, allow us to see a dashboard of tests that have passed and the ones that need fixing. When it comes to JavaScript Unit-Testing frameworks aimed at the DOM API, we need a test-framework that can
  1. simulate browser behavior such as mouse-hover, clicks, drag and drop
  2. allow us to control the test runner interactively
  3. give us the ability to filter what tests are to be executed
  4. provide a way to test asynchronous operations by grouping all those that depend on the same asynchronous operation
Name a few testing frameworks for JavaScript.
Answer: following are some of the popular JavaScript Testing Frameworks
  • Jasmine
  • mocha with chai
  • QUnit
  • JsUnit

Explain how functions work in JavaScript
Answer:
  • JavaScript functions can be invoked in one of four ways.
    1. as a function
    2. as a method on an object
    3. as an object constructor
    4. as apply/call
  • Every function is implicitly provided with two additional parameters, namely "this" and "arguments". The "this" parameter is influenced by above the invokation patterns (listed above) used
  • Additionally, JavaScript functions are basically objects (mutable key-value pairs) and are instances of the core JavaScript object named "Function" Therefore every function has a hidden link to "Function.prototype". Which means that anything attached to the Function.prototype instantly becomes available to all instances the functions.
  • Every function has a property called "prototype" which is an object consisting of a property called "constructor" whose value is the function itself.
  • functions can be stored in variables, objects, arrays, they can be passed to other functions as arguments and returned from other functions as values.

    How are parameters handled by functions?
    Answer: If a function is called by passing in more arguments than it can take, the extra arguments are ignored and if a function is called with fewer arguments than it takes, the called function proceeds with the execution by sticking in "undefined" for those parameters that didn't receive any value.
    Can we define var-args functions?
    Answer: We can achieve this by leveraging the implicitly passed bonus parameter called "arguments". It is basically an array-like object which holds the parameters passed to it as arguments from the calling function.
    How can we define a variable’s type?
    Answer: As JavaScript is a dynamically-typed language, the variable's type is determined by the type of value we assign to it
    How are variables declared?
    Answer: variables are declared with a "var" keyword;
    What is the purpose of the var keyword?
    Answer: "var" keyword is used to declare a variable. In ES5, the use of var keyword for a variable inside a function, influences it's scope. When "use strict" is not used, forgetting to use the var keyword for a variable intended as a local variable will cause to silently be scoped to its outer namespace.
    Describe briefly how JavaScript code is executed on the browser.
    Answer: Within the browser, the JavaScript code is interpreted and executed by the JavaScript Runtime Engine. Although we can author and execute JavaScript code without involving the DOM API, it would be meaningless to do so if we don't. When coding against the DOM API, the code we author is hooked in as eventhandlers. These eventhandlers are nothing but callback functions that are queued in on the "eventloop" (that runs on a single-thread) and executed in a first-come-first-served basis. The thread that registers events on the "eventloop" is separate from the one on which the eventloop itself runs.
